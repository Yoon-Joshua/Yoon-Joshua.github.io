Real-time Global Illumination

发表于2023-08-11|更新于2023-08-11|GAMES202

The global illumination technique usually is very costly usually. In the field of real-time rendering, people seek simple and fast solutions to one bounce indirect illumination. Remember that we assume that the light can bounce up to two times, one is direct illumination and the other is indirect illumination.

Like the following scene, the ray hits Q and is reflected to P, and is reflected to the camera. Q and its surface is lit directly. Any directly lit surface will act as a light source again. We call them reflectors.

Reflective Shadow Maps

Which surfaces are lit directly? A shadow map can tell us. Each texel in the shadow map is corresponding to a illuminated patch in the scene. Therefore each texel can be considered as a reflector.

How do these reflectors reflect lights? For simplicity, we believe that any reflector is diffuse, so that the reflectors reflect lights uniformly in all directions.

What about the visibility? Let’s ignore it. In practice, the visibility here can be ignored reasonably, because it will bring a high overhead if we compute the visibility for every pixel light and shading point. What’s more, most of high frequency information has been lost when the second bounce occurs, so the visual effect is satisfactory without the visibility.

Even so, the overhead is still high, because every texel in the shadow map is a reflector. To accelerate, we prefer to select a few reflectors to compute. We can select the reflectors around the shading point in the shadow map, after all the remote light sources have little influence on the shading point.

In general, we can say that the distance between xxx and a pixel light xpx_pxp​ in the shadow map is a reasonable approximation for their distance in world space. If the depth values with respect to the light source differ significantly, the world space distance is much bigger and we thus overestimate the influence. However, the important indirect lights will always be close, and these must also be close in the shadow map.

So we decided to obtain the pixel light samples as follows: first, we project x into the shadow map (s,t)(s,t)(s,t). We then select pixel lights around (s,t)(s,t)(s,t), where the sample density decrease with the squared distance to (s,t)(s,t)(s,t). This can easily be achieved by selecting the samples in polar coordinates relative to (s,t)(s,t)(s,t), i.e. if ξ1\xi_1ξ1​ and ξ2\xi_2ξ2​ are uniformly distributed random numbers, we select the pixel light at position

(s+rmaxξ1sin⁡(2πξ2),t+rmaxξ1cos⁡(2πξ2))(s+r_{max}\xi_1\sin(2\pi\xi_2),t +r_{max}\xi_1\cos(2\pi\xi_2)) (s+rmax​ξ1​sin(2πξ2​),t+rmax​ξ1​cos(2πξ2​))

We then have to compensate the varying sampling density by weighting the achieved samples with ξ12\xi_1^2ξ12​ (and a final normalization). An example sampling pattern is shown in the figure.

Well, this topic is supposed to be over now. In summary, what RSM method requires is with every texel the depth value, the world space position, the normal and the radiant flux.

Light Propagation Volumes

First, find directly lit surfaces. It’s sufficient to simply apply RSM.

Pre-subdivide the scene into a 3D grid. For each grid cell, find enclosed virtual light sources. Sum up their directional radiance distribution.

For each grid cell, collect the radiance received from each of its 6 faces. Sum up and again use spherical harmonics to represent. Repeat this propagation several times till the volume becomes stable.

For and shading point, find the grid cell it is located in. Grab the incident radiance in the grid cell from all directions. Then, shade.

Steps:

  1. Generation of radiance point set scene representation
  2. Injection of point cloud of virtual light sources into radiance volume
  3. Volumetric radiance propagation
  4. Scene lighting with final light propagation volume

LPV has a drawback. Like the following figure, there’s a thin object of which two sides are in the same grid. In reality, light from point ppp can’t reach the other side. But in the implement of LPV, light from one side could reach the other side because they are in the same grid.

Voxel Global Illumination

It’s still a two-pass algorithm.

At the pre-process stage, voxelize the entire scene and build hierarchy.

In the first pass, store the incident and normal distributions in each voxel and update on the hierarchy.

In the second pass, for glossy surfaces, trace 1 cone toward the reflected direction.

For diffuse, trace several cones.

Then query the hierarchy based on the (growing) size of the cone.

Screen Space Ambient Occlusion

SSAO is an approximation of global illumination in screen space, and cheap to implement.

In the theory of Blinn-Phong, ambient light is a constant light illuminating all of objects in the scene. What the SSAO will do is to consider different visibility towards all directions at different shading points.

The left is ambient term from Blinn-Phong and the right is ambient occlusion.

How to do it?

Limit to local occlusion in a hemisphere of radius RRR. Use the readily available depth buffer as an approximation of the scene geometry. Take samples in a sphere around each pixel and test against buffer. If more than half of the samples are inside, AO is applied, amount depending on ratio of samples that pass and fail depth test. We uses sphere instead of hemisphere, since normal information isn’t available.

Approximation of the scene geometry, some fails are incorrect. The one behind the red line for example.

Samples are not weighted by cos⁡θ\cos\thetacosθ, so not physically accurate, but looks convicing.

To improve the method above further, we approximate ray-tracing the depth buffer, which requires that the normal is known and only samples in a hemisphere. This new method is called horizon based ambient occlusion, HBAO for short.

Screen Space Directional Occlusion

Compare SSDO with SSAO.

In SSAO, we emit rays from the shading point. If a ray doesn’t intersect with anything, it means that the ambient light can illuminate the shading point from this direction. Otherwise, the intersection point is considered to occlude the shading point, which leads to shadows.

In SSDO, we emit rays from the shading point two. If a ray doesn’t intersect with anything, it means there’s no indirect illumination in this direction. Otherwise, the intersection point is considered as a reflector illuminating the shading point.

As we all know, it’s impossible for real ray-tracing in screen space, so we should adopt the method of screen space to approximate. For the shading point PPP, we randomly select NNN samples in a local hemisphere. In the following figure for example, there are 4 sample points, A, B, C and D.

Left: For direct lighting with directional occlusion, each sample is tested as an occluder. In the example, point P is only illuminated from direction C. Right: For indirect light, a small patch is placed on the surface for each occluder and the direct light stored in the framebuffer is used as sender radiance. In short, point P is lit directly from direction C, and is lit indirectly from the corresponding patches of occluders, A, B and D.

SSDO can correctly display the resulting colored shadows, whereas SSAO simply displays a grey shadow at each location. But in the third and fourth case of the following figure, the yellow shadow dispears. This is screen space issue: missing information from unseen surfaces. What’s more, the sampling hemisphere’s radius is finite, so it works in a short range.

Screen Space Reflection

The key idea of SSR is ray-tracing of one bounce in screen space. For each fragment, trace along ray direction using depth buffer, and then use the color of intersection point as reflection color.

But how to intersect in screen space. We can’t use the method of 3D space. What we have are depth buffer, color buffer, position buffer, normal buffer. Figuratively speaking, we have the surface of the whole scene, just like the skin.

The method of intersection in SSR is as following. We can get the direction of reflection easily if the reflection is specular. If the reflection is diffuse, we need to emit multiple rays for sampling. For simplicity, we just discuss the specular case. Along the reflected direction, the photon moves forward step by step until it arrives a position that’s behind the surface. To test if it’s behind the surface, we check depth values. The intersection point is between the position and the previous position.

How do we select the stride when the photon moves forward? We make the stride adaptive. Build a hierarchy, in other words, generate a depth Mip-Map using minimum values instead of average.
